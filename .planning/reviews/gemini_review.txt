### Launch-Blocking Adversarial Review: Jarvis Engine

**Final Gate: ❌ FAIL**

### Critical Severity Findings

#### C1: Path Traversal / Symlink Attack in `mobile_api.py`
*   **File:Line:** `engine/src/jarvis_engine/mobile_api.py:97` (`_gaming_state_path`) and `engine/src/jarvis_engine/mobile_api.py:126` (`_write_gaming_state`)
*   **Break Scenario:** The server constructs file paths by joining the repo root with fixed strings but does not validate the final resolved path. An attacker with local access (or who exploits another vulnerability to create a file) can create a symlink at `.planning/runtime/gaming_mode.json` pointing to a sensitive system file (e.g., `C:\Windows\System32\drivers\etc\hosts`). Sending a `POST /settings` request via the mobile API will then overwrite that file with JSON data, leading to system corruption or privilege escalation.
*   **Concrete Fix:** Use `.resolve()` on the path and verify it starts with the resolved `repo_root`.
    ```python
    def _gaming_state_path(self) -> Path:
        root = self.server.repo_root.resolve()
        target = (root / ".planning" / "runtime" / "gaming_mode.json").resolve()
        if not str(target).startswith(str(root)):
            raise PermissionError("Path traversal attempt blocked.")
        return target
    ```

#### C2: Non-Functional Autopilot (Logic Break)
*   **File:Line:** `engine/src/jarvis_engine/life_ops.py:134` (`export_actions_json`)
*   **Break Scenario:** The autopilot planner generates action records with an empty `"command": ""` field. The `AutomationExecutor` skips any action with an empty command string. Consequently, while Jarvis correctly identifies "Urgent tasks" or "Due bills," it never actually performs any work during its autopilot cycles, rendering the core value proposition of the daemon loop broken.
*   **Concrete Fix:** Implement a basic command generator or template system in `life_ops.py` to populate the `command` field with actionable CLI calls (e.g., `jarvis-engine log ...` or specific connector scripts).

### High Severity Findings

#### H1: Cleartext Transmission of Sensitive Credentials
*   **File:Line:** `engine/src/jarvis_engine/mobile_api.py:311` (`run_mobile_server`)
*   **Break Scenario:** The `ThreadingHTTPServer` used for the mobile API does not support TLS. All requests, including those containing the `Authorization` Bearer token and the `X-Jarvis-Signature` (derived from the signing key), are transmitted in plain text over the network. On a typical WiFi network used by a mobile device, these can be intercepted by a packet sniffer, allowing an attacker to hijack the Jarvis engine and control the user's system.
*   **Concrete Fix:** Wrap the server socket with `ssl.wrap_socket` or recommend a reverse proxy (like Nginx) with TLS termination. At minimum, the documentation must scream about this risk.

#### H2: Fragile Voice Intent Mapping
*   **File:Line:** `engine/src/jarvis_engine/main.py:656-780` (`cmd_voice_run`)
*   **Break Scenario:** The system maps voice-to-text strings to intents using simple `any(k in lowered for k in [...])` checks. This is highly susceptible to "negative" injection. For example, the command "Jarvis, do NOT pause the daemon" contains "pause daemon," which triggers the `runtime_pause` intent, achieving the exact opposite of the user's request.
*   **Concrete Fix:** Use a more robust NLP approach or at least check for negation keywords and ensure that keywords match specific command structures rather than simple substrings.

#### H3: Spoofable Voice Authentication
*   **File:Line:** `engine/src/jarvis_engine/voice_auth.py:111-137` (`_extract_embedding`)
*   **Break Scenario:** The voice embedding logic relies on extremely basic statistics (FFT mean/std, Zero Crossing Rate, RMS). This does not capture deep biometric features and is trivial to spoof with a basic recording or a low-quality synthesized voice. Since voice verification is used to gate "privileged" commands in `voice-run`, an attacker can easily bypass security by playing back a recorded tone or a person's voice.
*   **Concrete Fix:** Integrate a proper biometric embedding model (e.g., Wav2Vec2 or ECAPA-TDNN) for voiceprint extraction.

### Medium Severity Findings

#### M1: Non-Atomic State Writes
*   **File:Line:** `engine/src/jarvis_engine/runtime_control.py:47` and `mobile_api.py:133`
*   **Break Scenario:** State files are written using `Path.write_text()`. If the process crashes or the system loses power during the write operation, the JSON file can be left partially written or corrupted. On the next start, `json.loads()` will fail, potentially causing a crash or a reset to default "unsafe" states.
*   **Concrete Fix:** Write to a temporary file in the same directory and use `os.replace()` for an atomic update.

#### M2: Memory/Performance Exhaustion in MemoryStore
*   **File:Line:** `engine/src/jarvis_engine/memory_store.py:32` (`tail`)
*   **Break Scenario:** The `tail` method reads the *entire* `events.jsonl` file into memory just to return the last few lines. As the system runs for weeks and the log grows to megabytes or gigabytes, this operation will become increasingly slow and memory-intensive, eventually leading to OOM (Out Of Memory) crashes.
*   **Concrete Fix:** Use `f.seek()` to read from the end of the file in chunks until the required number of lines is found.
