<objective>
Replace the hand-rolled ICS parser with icalendar + recurring-ical-events for proper RFC 5545 support (RRULE expansion, VTIMEZONE, property folding), and add a pluggable task source layer that normalizes tasks from local JSON, Todoist API, or Google Tasks into a consistent format.

Purpose: The existing `_parse_ics()` handles only flat VEVENT blocks and ignores recurring events, timezones, and property folding. Real Google Calendar ICS exports contain all of these. The task connector currently reads only from a local JSON file -- adding a task source abstraction enables real task integrations.

Output: Upgraded `ops_sync.py` with proper ICS parsing and task source abstraction, updated `connectors.py` with task source env vars, new dependencies in `pyproject.toml`, and comprehensive tests.
</objective>

    2. Replace the existing `_parse_ics()` function in `engine/src/jarvis_engine/ops_sync.py` with a proper implementation:
       - Import `from icalendar import Calendar` and `import recurring_ical_events` at top of file (lazy import inside the function to avoid import errors if library not installed).
       - New `_parse_ics(text: str, target_date: date | None = None) -> list[dict]`:
         - Use `Calendar.from_ical(text)` to parse the ICS data.
         - Use `recurring_ical_events.of(cal).between(start, end)` to expand recurring events for `target_date` (default: `date.today()`). The range is `target_date` to `target_date + timedelta(days=1)`.
         - For each event, extract: `SUMMARY` (-> title), `DTSTART` (-> time as "HH:MM" or "all-day"), `LOCATION` (-> location), `DESCRIPTION` (-> description, truncated to 200 chars).
         - Sort events by time string.
         - Return list of dicts matching existing format: `{"title": ..., "time": ..., "location": ..., "prep_needed": "yes"}`.
         - Wrap the entire body in try/except: if icalendar or recurring-ical-events is not installed, fall back to the old line-by-line parser (keep old logic as `_parse_ics_fallback(text)`).
       - Update `load_calendar_events()`:
         - Accept optional `target_date: date | None = None` parameter.
         - Pass `target_date` to `_parse_ics()` calls for ICS file and ICS URL paths.
         - All existing env var checks remain unchanged: JARVIS_CALENDAR_JSON, JARVIS_CALENDAR_ICS_FILE, JARVIS_CALENDAR_ICS_URL.
         - Keep existing security checks: `_is_safe_calendar_url()`, `JARVIS_ALLOW_REMOTE_CALENDAR_URLS`, `MAX_ICS_BYTES`.

    3. Add task source loading to `ops_sync.py`:
       - New function `load_task_items(repo_root: Path) -> list[dict]`:
         - Read `JARVIS_TASK_SOURCE` env var (default: "json").
         - If "todoist": call `_load_todoist_tasks()` (lazy import todoist_api_python).
         - If "google_tasks": return empty list with a TODO comment (Google Tasks requires OAuth2, deferred to future).
         - Else (default "json"): read from `JARVIS_TASKS_JSON` env var path, or fallback to `repo_root / ".planning" / "tasks.json"` using existing `_read_json_list()`.
       - `_load_todoist_tasks() -> list[dict]`:
         - Read `JARVIS_TODOIST_TOKEN` env var. If empty, return [].
         - Lazy import `from todoist_api_python.api import TodoistAPI`.
         - Call `api.get_tasks(filter="today | overdue")`.
         - Normalize each task to: `{"title": t.content, "priority": {4:"urgent",3:"high",2:"normal",1:"low"}[t.priority], "due_date": t.due.date if t.due else "", "status": "pending"}`.
         - Wrap in try/except, return [] on any failure.

    4. Update `build_live_snapshot()` to use `load_task_items(root)` instead of `_read_json_list(planning / "tasks.json")` for task loading. Keep the existing variable name `tasks` so nothing downstream breaks.

    5. Run `pip install -e ".[dev]"` to verify the new dependencies install correctly.
