# Plan 05-01: Mobile App Readiness

## Requirements Covered
- MOB-01: Mobile-desktop sync verified end-to-end
- MOB-02: Mobile API endpoints tested for reliability
- MOB-03: Learning sync — mobile interactions contribute to desktop learning trackers
- MOB-04: Offline command queue verified
- MOB-05: Android app build verified against current desktop API surface

## Current State
- Desktop sync engine fully implemented (changelog triggers, compute_outgoing, apply_incoming, Fernet transport)
- Mobile API has 15 GET + 12 POST endpoints with HMAC auth + security pipeline
- Learning happens through /command endpoint (implicit via LearnInteractionCommand)
- Learning tables (user_preferences, response_feedback, usage_patterns) NOT in sync changelog
- Android only calls: /health, /cert-fingerprint, /bootstrap, /command, /settings, /dashboard
- Android does NOT call /sync/pull, /sync/push — bidirectional sync is server-ready but unused
- Offline queue in CommandQueueProcessor: queue, flush, retry (max 5), claim-for-send atomic

## Approach
Focus on the Python desktop side (we cannot build Android). Make the desktop API
surface fully ready for mobile deployment. All verification through Python tests.

## Tasks

### MOB-01: Add Learning Tables to Sync Changelog
**File**: `engine/src/jarvis_engine/sync/changelog.py`
**Change**: Add `user_preferences`, `response_feedback`, `usage_patterns` to `_TRACKED_TABLES`
**Why**: Learning data is desktop-only because these tables aren't tracked. Adding them enables
bidirectional learning sync once Android wires /sync/pull and /sync/push.
**Test**: Verify changelog triggers are installed for learning tables, INSERT/UPDATE/DELETE create entries.

### MOB-02: Add /learning/summary Endpoint
**File**: `engine/src/jarvis_engine/mobile_api.py`
**Change**: New GET /learning/summary endpoint that returns:
- Top preferences (from PreferenceTracker)
- Route quality scores (from ResponseFeedbackTracker)
- Peak usage hours (from UsagePatternTracker)
**Why**: Android needs a way to display learning insights. Currently no endpoint for this.
**Auth**: HMAC required.
**Test**: Verify endpoint returns correct structure.

### MOB-03: Add /feedback Endpoint
**File**: `engine/src/jarvis_engine/mobile_api.py`
**Change**: New POST /feedback endpoint accepting `{interaction_id, quality, comment}`.
Dispatches existing feedback mechanisms through ResponseFeedbackTracker.
**Why**: Android has no way to send explicit feedback (thumbs up/down). Only implicit detection exists.
**Auth**: HMAC required. Rate-limited: 10 req/min (expensive bucket).
**Test**: Verify endpoint processes feedback and updates tracker.

### MOB-04: Sync Protocol End-to-End Tests
**File**: `engine/tests/test_mobile_readiness.py`
**Tests** (~30 tests):
- Sync round-trip: desktop creates memory → /sync/pull returns it → simulated mobile pushes → /sync/push applies
- Learning table sync: preference created → changelog entry exists → pulled → pushed back
- KG sync: add_fact → changelog → pull → push
- Conflict resolution: overlapping field updates → desktop wins ties
- Offline queue simulation: queue N commands → flush → all execute → learning trackers updated
- API surface check: every Android-used endpoint responds with correct schema
- Command reliability: concurrent /command requests, malformed input, oversized body, timeout
- /feedback endpoint: explicit feedback flows to ResponseFeedbackTracker
- /learning/summary: returns valid structure with real data
- Auth endpoints: /auth/login returns session, /auth/logout invalidates
- Rate limiting: expensive endpoints throttled correctly
- HMAC edge cases: expired timestamp, reused nonce, invalid signature

### MOB-05: API Surface Compatibility Check
**File**: `engine/tests/test_mobile_readiness.py`
**Tests**: For each endpoint in JarvisApi.kt, verify desktop returns correct response shape:
- GET /health: {status, intelligence_regression_detected}
- POST /bootstrap: {base_url, token, signing_key, device_id}
- POST /command: {ok, command_exit_code, intent, status_code, reason, stdout_tail}
- GET /settings: {runtime_control, gaming_mode, owner_guard}
- GET /dashboard: contains jarvis_score, rankings, memory_regression keys
- GET /cert-fingerprint: {fingerprint}
