# Plan 14-09: Always-On Learning, Trust-Layered Promotion, and Zero-Regression Security

**Phase:** v5.0 Phase 6 (Security Expansion + Release Gate)  
**Execution model:** GSD (Goal -> Strategy -> Delivery)  
**Date:** 2026-03-11

## Requirements Covered

- SEC-02, SEC-03, SEC-04, SEC-06
- LM-05, LM-06
- REL-03, REL-04
- MOB-09, MOB-12

## Non-Negotiable Invariant

Jarvis learning must remain always-on. Security work must never disable learning as a product function.

What changes is not whether Jarvis learns, but how learned material is classified, promoted, retrieved, and allowed to drive side effects.

## User-Critical Outcomes

1. Preserve learning 100% of the time for preferences, corrections, habits, success/failure patterns, and intelligence growth.
2. Prevent untrusted external artifacts from becoming trusted reusable memory by default.
3. Add deterministic policy gates for execution, persistence, installation, and raw artifact promotion without weakening normal reasoning.
4. Improve security posture without causing regression in task success, recall quality, or mission throughput.
5. Make learning provenance visible and auditable across desktop, mobile, and long-running missions.
6. Ensure mobile-side learning/sync paths cannot silently lose data or duplicate actions under concurrency.

## Problem Statement

Current behavior mixes multiple trust classes inside one learning surface:

- semantic learning (preferences, corrections, habits)
- observed external material (downloaded files, copied code, shell snippets)
- action-driving memory and mission context

That creates three risks:

1. malicious or intrusive artifacts can be normalized into trusted memory
2. high-risk actions can fail soft or look successful
3. concurrency/sync bugs can silently lose learned state or duplicate side effects

The fix is not to reduce learning. The fix is to split learning into lanes and add promotion controls.

## Strategy

1. **Protect the perimeter, not the brain first:** keep reasoning broad; constrain only side effects and trust promotion.
2. **Never block observation:** Jarvis may always inspect, summarize, classify, and learn from new material.
3. **Gate promotion, not ingestion:** raw external artifacts enter quarantined/observed memory before they can influence trusted execution or retrieval.
4. **Use shadow mode before enforcement:** run the policy engine in audit-only mode first and prove low false-positive rates.
5. **Dual-write before cutover:** add provenance/trust metadata alongside current learning paths before changing retrieval or actuation.
6. **Preserve rollback:** every new policy path ships behind flags with warn-only and approval-only modes before hard block.

## Design Principles

1. **Learning is sacred; promotion is conditional.**
2. **Semantic summaries are safer than raw payload reuse.**
3. **Trust is earned through provenance, verification, and repeated safe outcomes.**
4. **The model may suggest broadly; the system may execute narrowly.**
5. **Deterministic policy must outrank model confidence for high-risk actions.**
6. **Suspicious content should enrich threat memory even when it never becomes trusted memory.**
7. **Every enforcement layer must be measurable, reversible, and correlation-traceable.**

## Scope Boundaries

### In Scope

- engine-side learning classification and promotion
- memory provenance and trust metadata
- mission/action policy classification
- retrieval trust weighting
- approval flows for high-risk actions
- Android/mobile sync integrity issues that can lose or duplicate learned state
- dashboard/activity surfaces needed to inspect trust and promotion

### Explicitly Out of Scope for This Plan

- replacing the LLM/router stack
- reducing provider flexibility or normal reasoning breadth
- disabling mobile learning or local knowledge growth
- introducing global refusal behavior for benign tasks
- deleting existing learning history as part of first rollout

## Assumptions

1. Jarvis must keep learning from user interactions continuously.
2. Conner is willing to approve narrowly-defined high-risk promotion/execution events, but not normal learning.
3. Current memory stores can be extended with metadata fields or companion tables without destructive migration.
4. Existing activity feed and mission telemetry can carry trust/policy events with correlation ids.
5. Android and engine can tolerate a dual-write transition period before retrieval cutover.

## Target Architecture

### Memory Lanes

1. **Trusted Learning Memory**
   - owner-confirmed preferences
   - repeated validated patterns
   - safe summaries
   - durable mission-relevant facts

2. **Observed Memory**
   - newly seen external content
   - provisional patterns
   - copied code and commands
   - downloaded artifacts awaiting verification

3. **Threat Memory**
   - malicious indicators
   - blocked hashes/domains
   - suspicious command chains
   - prior unsafe promotion attempts

### Promotion States

- `observed`
- `quarantined`
- `verified`
- `trusted`
- `blocked`

### Verification Sources

- owner approval
- local safety scanner result
- trusted-origin allowlist
- repeated benign-use evidence
- signed/hash-matched artifact evidence
- explicit policy exception

### Risk Classes

- `read`
- `write`
- `network`
- `execute`
- `install`
- `persist`
- `learn_from_artifact`

Only the last four may require approval or blocking. Normal reasoning and semantic learning do not.

### Trust Inheritance / Taint Rules

- Any output synthesized while `T1`, `T2`, or `T4` material is present in active context inherits the lowest trust level seen in that reasoning chain until verified.
- Summaries generated from quarantined artifacts are tagged with the source artifact trust level plus a `derived_from_artifact=true` marker.
- Policy review must inspect both direct artifact use and derived-action proposals so low-trust content cannot bypass controls by being rephrased by the model.
- Trust inheritance data must be traceable via `correlation_id`, `parent_memory_ids`, and `derived_from_action_id`.

## Threat Model

### Primary Failure Modes

1. raw external code or artifacts become trusted memory and later get re-used
2. malicious commands are learned as successful patterns because they once executed
3. copied snippets from external sources get promoted without provenance
4. mission retries or mobile replay produce duplicate side effects and misleading "success"
5. learned facts are silently lost because sync marks them complete before acknowledgement
6. mixed-trust retrieval causes low-trust artifacts to contaminate mission planning
7. broad error swallowing makes unsafe promotion/execution look like benign degradation

### Defended Channels

- web fetch and research ingestion
- mobile `/command` and sync ingestion
- task/misson artifact storage
- local file attachments and exports
- copied shell commands and generated scripts
- model/provider outputs that could be re-ingested as learning material
- low-trust context injection through summaries, reflections, or derived action drafts

## Trust Taxonomy

### Trust Levels

- `T0_untrusted`: unknown external origin, unverifiable, or suspicious
- `T1_observed`: recorded but not verified; okay for analysis only
- `T2_verified`: validated source and/or scanner passed; still not necessarily executable
- `T3_trusted`: owner-approved or repeatedly validated for bounded reuse
- `T4_blocked`: known bad; visible only for threat memory and forensics

### Promotion Rules

- `T0 -> T1`: automatic on observation
- `T1 -> T2`: allowed after provenance + verification checks
- `T2 -> T3`: allowed only through explicit promotion logic, typically owner approval or repeated safe validation
- `any -> T4`: immediate on known-malicious indicators or policy verdict

### Retrieval Behavior by Trust

- `T3`: eligible for normal retrieval and action-driving use
- `T2`: eligible for normal reasoning, limited action suggestions, no blind execution reuse
- `T1`: explainable/inspectable only, down-ranked in retrieval, excluded from automatic actuation
- `T4`: excluded from normal reasoning context except security/threat workflows

## Command and Mission Policy Contract

Every high-risk action proposal must be normalized into a deterministic policy envelope:

- `action_id`
- `mission_id`
- `correlation_id`
- `risk_class`
- `target_paths`
- `network_targets`
- `requires_approval`
- `would_modify_repo`
- `would_modify_pc_state`
- `would_execute_external_artifact`
- `rollback_plan`
- `verification_plan`

The model may propose action content, but the policy engine owns the final classification.

## Tamper Evidence and Trust Integrity

- Trusted promotions (`T3`) must be recorded in a tamper-evident log signed with a local key.
- Preferred implementation:
  - hardware-backed key where platform support exists
  - software-backed local key with rotation support as fallback
- The first rollout does not require signing every row in primary memory tables; it does require signing promotion events and trust-state transitions so out-of-band trust escalation can be detected.
- Any mismatch between row state and signed promotion history must downgrade the record to `T1_observed` or `T4_blocked` pending review.

## Data Lifecycle

- Quarantined raw artifacts must not live forever by default.
- Default retention:
  - raw quarantined artifacts: 30 days unless pinned by owner or active mission
  - derived semantic summaries: retained under normal memory policies
  - blocked threat indicators: retain longer for security analytics
- Retention expiry must not delete provenance needed to explain prior decisions.

## Approval Scope Model

- Support bounded approvals to reduce fatigue without creating blanket trust:
  - single action approval
  - hash/version-bound approval
  - path-bound approval
  - short time-bound approval
- Default policy: narrowest scope wins.
- Approval scopes must never silently generalize from one raw artifact to arbitrary future artifacts.

## Mission Pause/Resume Contract

- Any mission awaiting approval must be able to:
  - serialize current step state
  - persist pending artifacts and trust context
  - sleep without losing intermediate results
  - resume after approval or denial
- Approval denial must preserve semantic learning and diagnostics while preventing unsafe promotion or execution.

## Delivery Tasks

### Task A: Learning Trust Model and Provenance Schema

- Introduce learning provenance fields for all newly learned items:
  - `source_type`
  - `source_uri`
  - `source_hash`
  - `artifact_kind`
  - `trust_level`
  - `promotion_state`
  - `scan_status`
  - `approved_by_owner`
  - `approved_at`
  - `correlation_id`
- Keep current learning writes intact; dual-write new metadata without changing retrieval behavior yet.
- Define explicit semantic-vs-artifact classification rules so semantic learning remains unrestricted.
- Add migration strategy:
  - forward-compatible schema additions
  - backfill defaults for existing memory
  - no destructive rewrite of existing learned records
- Add explicit semantic categories that always remain eligible for automatic learning:
  - preferences
  - corrections
  - route quality
  - usage/habit patterns
  - user-confirmed facts
  - mission outcomes

### Task B: Observed Artifact Quarantine Lane

- Add a quarantine store for raw external artifacts, commands, snippets, and downloaded files.
- Learn safe semantic summaries immediately, but do not auto-promote raw artifacts into trusted memory.
- Record hashes, origin, MIME/type, and source channel.
- Allow Jarvis to reason about quarantined content without executing or reusing it by default.
- Define artifact classes requiring quarantine by default:
  - scripts
  - shell commands from external text
  - copied code blocks
  - archives
  - executables
  - macros/documents with active content
  - downloaded installers
- Define a safe-summary extraction path:
  - summarize intent
  - extract risk indicators
  - capture source and hash
  - store summary in observed/trusted semantic memory without promoting raw payload
- Safe-summary extraction must run in an isolated summarization path with a hardened prompt/policy contract so artifact content cannot directly turn into executable instructions without taint markers.

### Task C: Deterministic Mission Policy Engine (Shadow Mode First)

- Add a policy layer for high-risk action classes:
  - `execute`
  - `install`
  - `persist`
  - `learn_from_artifact`
- Initial mode is `audit_only`:
  - log what would have been blocked
  - log what would have required approval
  - log whether task outcome would differ
- No user-facing blocking in this task unless policy is already trivially safe.
- Add policy decision reasons:
  - unknown origin
  - executable artifact
  - persistence mutation
  - unsafe network target
  - repo mutation outside allowed scope
  - blocked prior indicator
- Add deterministic allowlists:
  - approved local directories
  - approved domains/endpoints
  - approved tool classes
  - approved file extensions for passive ingestion
- Add emergency revocation path:
  - downgrade `T3` to `T4`
  - invalidate downstream derived promotions
  - emit operator-visible "glass break" event

### Task D: Promotion Gates and Approval Contracts

- Add promotion rules:
  - semantic learning -> may promote automatically
  - artifact learning -> never auto-promote to trusted without verification
- Require explicit approval only for:
  - executing learned external commands/scripts
  - storing raw external artifacts as trusted reusable patterns
  - persistence/startup/service/task-scheduler style changes
  - package installation or remote code fetch that mutates the PC
- Approval payload must include:
  - exact action
  - target paths
  - source/provenance
  - risk class
  - rollback/test plan
- Approval decisions must be stored as auditable, replayable records tied to:
  - owner identity
  - time
  - artifact hash/version
  - mission/action ids
  - exact approved command or promotion
- Approval must not auto-whitelist all future similar actions unless explicitly configured.
- Add fatigue controls:
  - support path-bound/time-bound approvals
  - track approval abandonment
  - recommend narrower or broader scope only from explicit owner action

### Task E: Trust-Aware Retrieval and Mission Use

- Keep all learning queryable, but rank by trust.
- Retrieval rules:
  - trusted > verified > observed
  - blocked never enters execution-driving retrieval
  - quarantined content may be visible for explanation but not auto-used in tool execution
- Add down-ranking before exclusion to avoid sudden recall regression.
- Preserve existing recall paths during shadow mode and compare outputs.
- Add retrieval A/B capture:
  - current retrieval result set
  - trust-weighted retrieval result set
  - delta report for regression analysis
- Add "why this memory was used" support to aid debugging and false-positive review.
- Add freshness override rule so a newly referenced user artifact can still be found for analysis, even if low trust, without allowing automatic execution reuse.

### Task F: Mobile Integrity Hardening for Learning/Sync Safety

- Make Android command queue flushing single-claim and idempotent to prevent duplicate sends.
- Ensure local facts are only marked synced after network success/ack, never before.
- Make `AutoSyncManager.start()` idempotent and callback-safe.
- Remove cleartext bootstrap path for any flow carrying master-password material.
- Validate remote sync intervals and retry settings before applying them.
- Add missing Android tests for command queue concurrency, sync acknowledgement, and bootstrap transport.
- Add engine-side tests covering matching server guarantees so mobile and desktop contracts stay aligned.
- Require an explicit ack protocol for learned fact export before `synced=true`.
- Eliminate duplicate queue processing windows by using claim-and-complete semantics rather than scan-and-send semantics.
- Add bounded offline queue behavior:
  - max queue sizes
  - eviction/compaction rules
  - exponential backoff for repeated network failures
- Add explicit trust conflict protocol for mobile/desktop divergence:
  - trust downgrade wins over promotion
  - blocked state propagates immediately
  - conflicts emit audit events instead of silently last-write-wins

### Task G: Evaluation Harness and Regression Gates

- Build four acceptance suites:
  1. normal learning tasks
  2. high-risk artifact tasks
  3. mixed safe/unsafe context tasks
  4. mission retry/concurrency/sync tasks
- Track:
  - task success rate
  - learning recall quality
  - security catch rate
  - false-positive rate
  - mission throughput
- Promotion from `audit_only` -> `warn_only` -> `approval_required` requires evidence that normal capability did not regress.
- Add mandatory benchmark classes:
  - benign daily-assistant tasks
  - learning-heavy correction tasks
  - repo analysis tasks with copied code context
  - malicious-payload simulation tasks
  - mixed-trust mobile sync tasks
- Define promotion thresholds:
  - no significant drop in benign task completion
  - no significant drop in semantic memory recall
  - measurable increase in unsafe promotion catch rate
  - acceptable operator review burden
- Add friction telemetry:
  - `time_to_approve`
  - `approval_abandoned`
  - `approval_scope_used`
  - `policy_disagreement_rate`
  - `taint_leakage_rate`

### Task H: Rollout, Telemetry, and Forensics

- Add flags for each enforcement layer:
  - `audit_only`
  - `warn_only`
  - `approval_required`
  - `hard_block`
- Log all high-risk decisions with correlation id and provenance chain.
- Add dashboard/activity surfaces for:
  - learning lane
  - trust level
  - promotion state
  - approval requests
  - blocked artifact attempts
- Add trend views for:
  - promotions by trust class
  - blocked attempts by channel
  - approval volume
  - false-positive review outcomes
  - shadow-policy disagreement rate

## Rollout Plan

### Stage 0: Planning and Schema Readiness

- finalize taxonomy
- add schemas/flags
- no behavior change

### Stage 1: Dual-Write Provenance

- all new learning writes include trust/provenance metadata
- current retrieval/actuation unchanged
- telemetry only
- promotion events signed in tamper-evident log

### Stage 2: Shadow Policy

- classify all high-risk actions
- emit `would_warn`, `would_require_approval`, `would_block`
- no blocking

### Stage 3: Trust-Aware Retrieval Ranking

- down-rank low-trust artifact-derived memories
- compare against baseline retrieval output
- keep rollback flag

### Stage 4: Approval-Required Promotion

- require approval for raw artifact promotion and high-risk execution reuse
- semantic learning unchanged

### Stage 5: Hard Block Known-Bad Paths

- block only clearly malicious or policy-prohibited artifact/persistence patterns
- continue learning threat indicators and summaries

### Stage 6: Scoped Approval Optimization

- enable bounded approval scopes after friction telemetry proves the base approval model is too noisy
- keep single-action approval as the secure default

## Failure and Rollback Strategy

1. Every stage guarded by feature flags.
2. If recall quality drops, revert retrieval weighting without disabling provenance capture.
3. If approval burden is too high, stay in `warn_only` while preserving telemetry.
4. If mobile sync changes regress reliability, revert to previous transport behavior while keeping idempotency tests.
5. Never roll back the threat-memory recording path unless it is corrupting normal learning.
6. If taint propagation proves over-broad, narrow its execution impact before reducing provenance capture.

## Zero-Regression Rules

1. No change may disable semantic learning.
2. No change may globally reduce reasoning scope or context access.
3. No enforcement change may ship before shadow telemetry exists.
4. No retrieval cutover may occur before dual-write provenance has passed regression comparison.
5. Any new security gate must have a config kill-switch.
6. Any trust-based suppression must first ship as ranking/down-weighting before hard exclusion unless the artifact is clearly malicious.
7. Shadow telemetry must prove the policy is not suppressing normal preference/correction learning.
8. Mobile-side hardening must preserve offline queueing and end-to-end learning delivery.
9. Taint tracking must not hide user-requested low-trust artifacts from analysis-only workflows.

## Open Questions to Resolve During Execution

1. Should trust metadata live inside existing memory rows, companion tables, or both?
2. What is the exact "trusted enough for automatic reuse" threshold for generated code versus downloaded code?
3. Which directories and file types are considered safe for passive learning by default?
4. How should threat-memory indicators decay, if at all?
5. Which approval actions should expire by hash/version versus remain durable?
6. How should model outputs be re-ingested when the model itself synthesized code from mixed-trust context?

## Verification Plan

1. Semantic learning continuity tests:
   - preferences still adapt
   - corrections still influence routing and prompts
   - habits and usage patterns still update
2. Artifact quarantine tests:
   - external payload learned as observed/quarantined
   - semantic summary learned immediately
   - raw artifact not promoted without approval
3. Retrieval regression tests:
   - trusted recall unchanged for normal tasks
   - suspicious content down-ranked but still inspectable
4. Mission policy shadow tests:
   - risky actions classified correctly
   - no blocking in audit-only mode
   - would-block telemetry emitted
5. Android reliability tests:
   - no duplicate queue flush side effects
   - synced flags set only after successful push
   - repeated `start()` calls do not duplicate callbacks
6. Security/forensic tests:
   - correlation ids preserved across learning, approval, and execution surfaces
   - approval-denied paths leave learned summaries intact but prevent promotion

## Exit Criteria

- Jarvis continues to learn semantic information with no observed loss in learning behavior.
- Raw external artifacts are never auto-promoted into trusted reusable memory.
- High-risk actions are policy-classified deterministically and audited before enforcement.
- Retrieval remains stable for normal tasks while suspicious content is trust-ranked appropriately.
- Android/mobile learning and sync paths are idempotent and no longer silently lose learned data.
- Approval contracts exist for high-risk promotion/execution paths.
- Shadow-mode telemetry proves low false positives before any enforcement step is enabled.

## Execution Notes

- This plan should begin in parallel with ongoing reliability work only in non-invasive shadow-mode tasks.
- Any enforcement cutover should wait until current Phase 1 reliability gates are green and the mobile concurrency/data-loss fixes are landed.
- This plan is intended to preserve Jarvis's core identity: always learning, always improving, but never trusting raw external artifacts by accident.
