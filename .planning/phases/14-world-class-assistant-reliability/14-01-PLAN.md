## Current Baseline

- Tests: `4433 passed, 9 skipped, 0 failed`
- Lint: `ruff clean`
- Typed quality debt: `mypy reports 116 errors in 35 files`
- Security scan backlog: `bandit reports 165 findings (1 high, 50 medium, 114 low)`
- Runtime pain from user reports:
  1. Breakage after 1-2 commands
  2. Context drops on long outputs / HTTP boundary failures
  3. Memory pressure and high intensity under extended use
  4. Activity/progress visibility not trustworthy enough

## Goal

Create a stable execution substrate where Jarvis can run continuously, fail gracefully, preserve
context under load, and expose accurate activity telemetry before adding more capability.

## Strategy

1. Reliability before new features.
2. Enforce measurable SLOs (soak stability, memory ceilings, failure rates).
3. Remove silent failures and hidden retries.
4. Make mission/activity state backend-truth-driven and observable.
5. Gate merges on regression evidence, not assumptions.

## Delivery Tasks

### Task A: Command Lifecycle Hardening

- Standardize command lifecycle (`accepted -> running -> completed|failed|cancelled`).
- Add correlation IDs across API logs, activity feed, and mission store.
- Ensure each error path returns structured failure payload:
  - `error_code`, `category`, `retryable`, `user_hint`, `diagnostic_id`.

### Task B: Context Transport Resilience

- Add response chunking/streaming for large outputs in mobile API and desktop surfaces.
- Introduce conversation checkpoints:
  - rolling compressed summary
  - key-entity memory anchors
  - mission-local working set
- On provider switch/fallback, reconstruct context from checkpoint + latest deltas.

### Task C: Resource Guardrails

- Define hard memory budgets for:
  - embedding caches
  - conversation buffers
  - mission working state
- Add adaptive throttling for background loops when memory/CPU exceed thresholds.
- Emit resource pressure events to activity feed and dashboard.

### Task D: Observability Upgrade

- Extend activity feed schema with:
  - `mission_id`, `provider`, `step`, `progress_pct`, `correlation_id`.
- Ensure progress bars are driven from backend completion counters.
- Add reliability panel to dashboard:
  - command success rate
  - retry counts
  - timeout counts
  - memory pressure incidents

### Task E: Debt Burn-Down Gate

- Triage and classify all current `mypy` and `bandit` findings:
  - true bug/security risk
  - accepted risk with rationale
  - tooling false positive with explicit suppression
- Set release thresholds for this phase:
  - no unsuppressed high severity security findings
  - no new mypy errors in touched files
  - no new flaky tests introduced
