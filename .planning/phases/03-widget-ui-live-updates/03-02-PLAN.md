# Plan 03-02: Widget Frontend — Live Activity Feed & Immediate Refresh

**Phase:** 3 (Widget & UI Live Updates)
**Requirements:** UI-01 through UI-05 (frontend integration)
**Depends on:** 03-01 (backend: cancel command, activity events, recent_events in /widget-status)

## Context

Plan 03-01 added all backend support: MissionCancelCommand, activity event logging,
recent_events in /widget-status, and response= output. This plan wires the widget
frontend to consume that data — immediate dashboard refresh, expanded learned indicator,
and live activity event display.

## Tasks

### Task 1: Immediate Dashboard Refresh After Commands
After `_send_command_async` completes a command that modifies brain state, trigger
an immediate `/widget-status` fetch to update growth labels. This eliminates the
8-second stale data window for mission cancel, brain_status, etc.

**Files:** `desktop_widget.py` — `_send_command_async` worker, after response parsing

### Task 2: Expanded Learned Indicator Trigger
Expand the `_show_learned_indicator()` trigger from 3 intents to include all
knowledge-modifying intents: mission_cancel, brain_status, fact_extracted,
preference_learned, harvest, mission_create, mission_run.

**Files:** `desktop_widget.py` — line ~2433

### Task 3: Live Activity Feed Display
In the health loop, extract `recent_events` from `/widget-status` and display new
events (not previously seen) in a dedicated activity line format. Use event_id
deduplication to avoid repeats. Show timestamped, categorized entries.

**Files:** `desktop_widget.py` — `_health_loop`, `_set_online`, new `_update_activity_events`

### Task 4: Tests
Add widget frontend logic tests for activity event parsing, deduplication, and
learned indicator trigger expansion.

**Files:** `engine/tests/test_ui_frontend.py`

## Verification
- All 4200+ tests pass
- Widget displays activity events from /widget-status polling
- Learned indicator fires for broader set of intents
- Dashboard refreshes immediately after brain-modifying commands
